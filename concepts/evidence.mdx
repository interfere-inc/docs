Evidence is the complete collection of signals that Interfere gathers to understand what is wrong with your application and why. Rather than leaving you to manually correlate error logs, session replays, deployment records, and code history, Interfere pulls all of these signals together automatically the moment it opens an issue. Everything Interfere uses to reach its conclusions is visible to you in the **Evidence** tab of the issue detail panel, giving you full transparency into its reasoning.

## Types of Evidence

Interfere collects several categories of evidence, each contributing a different dimension to its understanding of a problem.

<CardGroup cols={2}>
  <Card title="Error Events" icon="circle-exclamation">
    Individual error records emitted by your application — exceptions, unhandled rejections, HTTP error responses. Interfere captures the stack trace, the affected endpoint or component, and the frequency of occurrence over time.
  </Card>
  <Card title="Session Data" icon="play">
    User session recordings and behavioral data that show exactly what a real user experienced before, during, and after the error. Sessions help confirm that the problem has a real user impact and not just a synthetic signal.
  </Card>
  <Card title="Code Diffs" icon="code-compare">
    Changes introduced in recent commits or pull requests that are connected to the affected code paths. Interfere surfaces the specific lines that changed, making it easy to spot an accidental regression.
  </Card>
  <Card title="Deployment Events" icon="rocket">
    Records of when new versions of your application were deployed. Interfere correlates the timing of deployments with the onset of anomalies to determine whether a release triggered the problem.
  </Card>
  <Card title="Log Entries" icon="scroll">
    Structured and unstructured log output from your application and infrastructure. Interfere filters the noise and surfaces only the log lines most relevant to the active issue.
  </Card>
</CardGroup>

## Facts

A **Fact** is an automatically surfaced, data-backed conclusion that Interfere derives from the raw evidence. Facts are precise, quantitative statements about what changed and when — they do not require interpretation. You can rely on them as ground truth extracted from your telemetry.

**Example fact from Issue #120:**

> *Error rate increased from 0.2% to 12% within 6 minutes after deployment.*

This single fact tells you three things at once: the baseline error rate before the problem, the peak error rate during the problem, and the tight causal window between a deployment and the degradation. Interfere surfaces facts like this in the Activity feed and the Evidence tab as soon as it computes them.

<Note>
  Facts are generated automatically — no configuration required. They appear in the Activity timeline marked with a **Fact** label so you can distinguish them from human comments and agent findings.
</Note>

## Findings

A **Finding** is an agent-generated insight that links one or more pieces of evidence to a likely root cause. Where a fact tells you *what* changed, a finding tells you *why* — connecting signals from error events, code diffs, and deployments into a coherent explanation.

**Example finding from Issue #120:**

> *A recent change in `auth/reset.ts` may cause invalid token validation.*

Interfere backs this finding with the specific code diff it identified:

```typescript
// auth/reset.ts — lines 7–14

export function buildResetLink(userId: string) {
  const token = sign<ResetTokenPayload>({ uid: userId });

- const baseUrl = "https://staging.example.com";
+ const baseUrl = "https://example.com/reset-password";

  const path = "/reset-password";
  const url = new URL(path, baseUrl);
  url.searchParams.set("token", token);
```

The finding connects the deployment event (a new release landed), the code diff (the `baseUrl` was changed in `auth/reset.ts`), and the error event (password reset flows are failing) into a single explanation: the reset link was pointing to a route that no longer exists after the deployment, redirecting users to a generic error page instead of the password reset form.

<Tip>
  When a finding references a file in your codebase, you can click **Open** directly from the finding to jump to that file. This works because Interfere is connected to your code repository via the Code Connection integration.
</Tip>

## How Facts and Findings Differ from Raw Evidence

It helps to think of these three layers as a pyramid:

<Steps>
  <Step title="Raw Evidence (base)">
    The unprocessed signals: individual error records, session replays, log lines, deployment records, and code diffs. These are the inputs to Interfere's analysis.
  </Step>
  <Step title="Facts (middle)">
    Derived, quantitative conclusions automatically computed from raw evidence. Facts summarize patterns — like an error rate spike — without requiring you to read through hundreds of individual records.
  </Step>
  <Step title="Findings (top)">
    Agent-generated causal explanations that connect multiple facts and evidence types to a root cause. Findings are the most actionable output: they tell you specifically what to look at and why.
  </Step>
</Steps>

## Evidence in the Issue Detail Panel

The **Evidence** tab on an issue surfaces the complete list of evidence items collected for that issue. Each item is labeled by type (error event, session, log entry, code diff, deployment) and sorted by relevance. You can expand any item to see the full detail.

The **Activity** tab interleaves facts and findings into the chronological story of the issue alongside human actions and status changes, so you can see exactly when Interfere reached each conclusion and what prompted it.

<Accordion title="What if I disagree with a finding?">
  Findings are Interfere's best assessment based on the evidence available at the time. If a finding points in the wrong direction, you can dismiss it and leave a comment explaining why. Interfere incorporates team feedback to improve future analysis. You can also add your own observations directly to the Activity feed.
</Accordion>
