An issue — also called a **Problem** in Interfere — is the central unit of work that represents something going wrong in your product. Rather than waiting for a user to file a support ticket or an engineer to notice a spike in a dashboard, Interfere continuously monitors your application and opens an issue the moment it detects an anomaly. From that point forward, everything Interfere learns about the problem is organized under a single issue record that your whole team can work from.

## How Issues Are Created

Interfere analyzes a continuous stream of signals from your application — error rates, session behavior, log patterns, deployment events, and more. When it detects a meaningful deviation from normal behavior, it automatically creates an issue and begins investigating. You never have to manually file a bug report for something Interfere has already seen.

<Note>
  Interfere creates issues proactively, before your users report them. The goal is for you to know about a problem — and often fix it — before it reaches your support queue.
</Note>

For example, when users began failing to complete password resets in production, Interfere detected a surge in failed attempts and immediately opened **Issue #120: Broken password reset link**, classifying it as High priority and associating it with the **Website Frontend** and **iOS App** surfaces.

## Issue Lifecycle

Every issue moves through a defined set of statuses that reflect where Interfere and your team are in the resolution process.

<Steps>
  <Step title="Detected">
    Interfere has identified an anomaly and opened the issue. Initial signals have been collected and the problem description has been generated. The issue appears in your inbox immediately.
  </Step>
  <Step title="Investigating">
    Interfere is actively gathering evidence, running analysis, and producing findings. The system is working to understand the root cause. Team members may also begin looking at the issue during this phase.
  </Step>
  <Step title="Active">
    The issue is confirmed and ongoing — real users are being impacted right now. The issue has been prioritized and assigned. Your team should be actively working toward a fix.
  </Step>
  <Step title="In Recovery">
    A fix has been deployed or a mitigation is in place. Interfere is monitoring whether the problem is subsiding. The issue remains open until signals confirm the problem is gone.
  </Step>
  <Step title="Resolved">
    The anomaly is no longer detected. Interfere has confirmed that error rates, session behavior, and other signals have returned to baseline. The issue is closed and archived for future reference.
  </Step>
</Steps>

## Issue Fields

Each issue exposes the following fields in the inbox row and the issue detail panel.

<CardGroup cols={2}>
  <Card title="Title" icon="heading">
    A plain-language description of the problem, auto-generated by Interfere from the detected signals. For example: *Broken password reset link*.
  </Card>
  <Card title="Priority" icon="flag">
    Reflects the severity and urgency of the issue. Interfere assigns **High**, **Medium**, or **Low** based on error rate, impacted user count, and surface criticality. You can override priority manually at any time.
  </Card>
  <Card title="Assignee" icon="user">
    The team member responsible for driving the issue to resolution. Interfere can auto-assign based on code ownership or surface routing rules, and any team member can reassign from the detail panel.
  </Card>
  <Card title="Status" icon="circle-dot">
    The current lifecycle stage of the issue: Detected, Investigating, Active, In Recovery, or Resolved. Status updates are logged in the issue activity feed.
  </Card>
  <Card title="Surface(s)" icon="layers">
    The area(s) of your product where the issue is observed — for example, **Website Frontend** and **iOS App**. An issue can span multiple surfaces when the same underlying problem affects more than one part of your product.
  </Card>
  <Card title="First Detection Time" icon="clock">
    The timestamp when Interfere first observed the anomaly. This anchors the issue timeline and helps you correlate the problem with recent deployments or other changes.
  </Card>
  <Card title="Duration" icon="timer">
    How long the issue has been active, calculated from first detection to the present (or to resolution). For example: *36 min 47 s*.
  </Card>
  <Card title="Impacted Users" icon="users">
    The number of unique users who have encountered the problem since first detection. This is one of the primary signals Interfere uses to determine priority.
  </Card>
  <Card title="Evidence Count" icon="magnifying-glass">
    The total number of evidence items — error events, sessions, log entries, code diffs, and deployment events — that Interfere has collected for this issue.
  </Card>
</CardGroup>

## Regression Flag

When Interfere determines that an issue was caused by a recent code change or deployment, it marks the issue as a **Regression**.

<Warning>
  A regression flag means this behavior previously worked correctly and was broken by a change your team introduced. Regressions are given higher investigative priority because their root cause is typically traceable to a specific commit or deployment.
</Warning>

Issue **#120** (Broken password reset link) is flagged as a regression. Interfere traced the failure to a change in `auth/reset.ts` that was introduced in the latest deployment, where the `baseUrl` was accidentally set to a staging domain instead of production. Because the password reset flow was working before this change, the issue is marked as a regression.

You'll see the **Regression: Yes** badge prominently in both the inbox row and the issue detail panel whenever this flag is set.

## The Inbox and Issue Detail Panel

### Inbox

The inbox is your team's real-time view of all open issues. Issues are grouped by status — **Under Investigation**, **Active**, **In Recovery** — so you can immediately see what needs attention. Each row shows the issue number, title, assignee avatar, priority badge, surface tags, and impacted user count at a glance.

You can filter the inbox by surface to focus only on the areas of the product your team owns. See [Surfaces](/concepts/surfaces) for more on filtering.

### Issue Detail Panel

Clicking any issue opens the detail panel, which is organized into three tabs:

<CardGroup cols={3}>
  <Card title="Activity" icon="timeline">
    A chronological feed of everything that has happened on the issue: when Interfere detected the problem, status changes, agent findings, facts, team member assignments, comments, and suggested fixes.
  </Card>
  <Card title="Sessions" icon="play">
    A list of every user session impacted by the issue, with full session context so you can replay exactly what a user experienced.
  </Card>
  <Card title="Evidence" icon="file-magnifying-glass">
    The complete collection of signals Interfere gathered — error events, log entries, code diffs, deployment records, and more. See [Evidence](/concepts/evidence) for a full breakdown.
  </Card>
</CardGroup>

<Tip>
  The Activity tab is the best place to understand the full story of an issue — it interleaves agent findings, human decisions, and external events like pull requests into a single readable timeline.
</Tip>
