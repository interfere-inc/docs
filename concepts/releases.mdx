Release tracking is how Interfere connects what your team ships to what breaks in production. Every deployment, pull request, and CI run that lands in your environment is a potential cause of a new issue. By ingesting release events alongside your application telemetry, Interfere can answer the question that matters most during an incident: *did something we just deployed cause this?*

## How Interfere Connects Deployments to Issues

When you integrate Interfere with your deployment pipeline and version control system, it begins receiving a stream of release events — a deployment completes, a pull request merges, a CI run finishes. Interfere correlates the timing of these events with the onset of anomalies in your application.

If an error rate spikes, a user flow starts failing, or a new pattern of exceptions appears shortly after a deployment, Interfere draws a connection between the two. This correlation is one of the primary inputs into both the regression detection system and the agent findings that appear in the issue timeline.

<Note>
  Interfere does not require code connection to detect issues. However, connecting your releases unlocks regression detection, code diff evidence, and findings that point to specific files and commits.
</Note>

## Regression Detection

When Interfere determines that an issue was introduced by a specific deployment, it flags the issue as a **Regression**. This tells your team that the behavior was working correctly before the change and that the fix likely involves reverting or patching a recent commit.

**How regression detection works:**

<Steps>
  <Step title="Baseline established">
    Interfere maintains a model of your application's normal behavior — error rates, latency distributions, flow completion rates — per surface and per code version.
  </Step>
  <Step title="Deployment event received">
    A new deployment is recorded. Interfere begins comparing incoming signals against the pre-deployment baseline.
  </Step>
  <Step title="Anomaly detected">
    If signals deviate meaningfully from baseline within a short window after the deployment, Interfere opens or updates an issue and begins investigating the deployment as a likely cause.
  </Step>
  <Step title="Regression flag set">
    If the investigation confirms that the anomaly correlates with the deployment — and no similar anomaly existed before it — Interfere marks the issue as a regression and surfaces the relevant code diffs as evidence.
  </Step>
</Steps>

For Issue **#120** (Broken password reset link), Interfere detected that the error rate on the password reset flow jumped from **0.2% to 12% within 6 minutes after a deployment**. It identified the deployment as the likely trigger and flagged the issue as a regression, directing the investigation toward the changes introduced in that release.

## The Timeline View

The issue timeline is a chronological record that interleaves every type of event relevant to the problem — agent findings, human actions, deployment events, pull requests, and CI runs — into a single readable view.

```
[4 min ago]  Problem detected — surge in failed password reset attempts
[4 min ago]  Status updated to Investigating
[3 min ago]  Finding — change in auth/reset.ts may cause invalid token validation
[3 min ago]  Fact — error rate increased from 0.2% to 12% within 6 min after deployment
[3 min ago]  Assigned to Luke Shiels
[3 min ago]  Priority set to High
[3 min ago]  Status updated to Active
[2h ago]     Jakub Krehel commented: "Looks related to yesterday's auth refactor. Investigating token validation."
```

This view is designed to answer a critical question during an incident: *what happened, in what order, and who or what did it?* Deployment and pull request events appear inline so you can see exactly where a release fits in the story of the problem.

<Tip>
  The timeline distinguishes between agent-generated findings (automated), human actions (team members), and external events (deployments, CI runs) with distinct visual treatments. This makes it easy to trace both the technical sequence of events and the human decisions made along the way.
</Tip>

## Code Connection and Root Cause Analysis

Code connection is the integration that gives Interfere access to your repository. With code connection enabled, Interfere can:

<CardGroup cols={2}>
  <Card title="Surface code diffs as evidence" icon="code-compare">
    When a finding points to a specific file or function, Interfere can pull the relevant diff directly into the evidence panel — showing you exactly what lines changed in the deployment that preceded the issue.
  </Card>
  <Card title="Link findings to commits" icon="code-branch">
    Findings reference the specific commit, pull request, or file that Interfere believes is responsible. You can open the file or PR directly from the finding without leaving Interfere.
  </Card>
  <Card title="Understand your product's structure" icon="sitemap">
    Interfere reads your codebase to understand how components relate to each other, improving the accuracy of root cause analysis and helping it navigate complex multi-service architectures.
  </Card>
  <Card title="Suggest targeted fixes" icon="wand-magic-sparkles">
    With access to the affected code, Interfere can propose a specific fix — not just a general direction — so engineers spend less time figuring out what to change and more time shipping the correction.
  </Card>
</CardGroup>

## Why Release Tracking Accelerates Resolution

Without release tracking, debugging a production issue means manually cross-referencing your deployment log against your monitoring dashboard, pulling up the relevant diff in your version control system, and working backward through recent changes to find the culprit. This process can take hours.

With release tracking connected, Interfere collapses this process automatically. The moment an issue opens, the relevant deployment is already linked, the code diff is already surfaced as evidence, and the finding already points to the specific file and function most likely responsible.
