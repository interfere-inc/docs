The Inbox is where you'll spend most of your time in Interfere. Every issue your application produces is automatically detected, triaged, and placed here — no tickets to file, no alerts to configure, and no manual creation required. From the moment Interfere detects a problem, it appears in the Inbox with everything your team needs to understand severity, ownership, and scope at a glance.

## Status groupings

Issues in the Inbox are organized into four status groups. These groups update in real time as Interfere's agent works and your team takes action.

<CardGroup cols={2}>
  <Card title="Under Investigation" icon="magnifying-glass">
    Interfere has detected the issue and its agent is actively gathering context — analyzing errors, sessions, deployments, and code changes to understand the root cause.
  </Card>
  <Card title="Active" icon="circle-exclamation">
    The issue is confirmed and ongoing. Users are currently being affected and the team is working toward a fix.
  </Card>
  <Card title="In Recovery" icon="arrow-up-right">
    A fix has been deployed or a mitigation is in place. Interfere is monitoring signals to confirm the issue is resolving.
  </Card>
  <Card title="Resolved" icon="circle-check">
    The issue is no longer affecting users and has been marked closed. Resolved issues are archived but remain searchable.
  </Card>
</CardGroup>

A typical Inbox might look like this:

| Status | # | Issue | Priority | Impacted Users |
|---|---|---|---|---|
| Under Investigation | #87 | Users unable to export billing invoices | High | 1,521 |
| Under Investigation | #65 | Payments failing for users in Brazil | Low | 340 |
| Active | #120 | Broken password reset link | High | 2,122 |
| Active | #108 | Onboarding emails not being delivered | Medium | 870 |
| Active | #95 | Search results showing wrong products | Low | 109 |
| In Recovery | #74 | Team invite links expiring too soon | Medium | 1,120 |
| In Recovery | #61 | Dashboard charts blank for new accounts | Low | 891 |

## Issue cards

Each entry in the Inbox is an issue card. Cards are designed to give you the most critical information without having to open the full detail view.

Every issue card shows:

- **Issue number** — a stable numeric identifier (e.g., `#120`) you can reference in comments, PRs, or external tools
- **Title** — a human-readable summary generated by Interfere (e.g., *Broken password reset link*)
- **Assignee avatar** — the team member currently responsible for the issue
- **Priority** — `High`, `Medium`, or `Low`, set automatically based on error rate and impacted user count, and adjustable by your team
- **Impacted user count** — the number of unique users affected since first detection
- **Status** — current lifecycle state, reflected by which group the card sits in

<Note>
  Issue cards update live. If Interfere detects that 500 more users have hit the broken password reset flow in the last two minutes, the impacted user count on card `#120` increments automatically — no refresh needed.
</Note>

## Filtering, sorting, and searching

The Inbox supports several ways to cut through noise and focus on what matters to you.

### Filtering

Use the filter bar above the issue list to narrow down by:

- **Surface** — show only issues affecting a specific part of your product (e.g., *Website Frontend*, *iOS App*, *API*)
- **Priority** — filter to `High`, `Medium`, or `Low` priority issues
- **Assignee** — view only the issues assigned to a specific team member, useful during on-call or sprint planning

Filters can be combined. For example, you can filter to all `High` priority issues on the `iOS App` surface assigned to a particular engineer.

### Sorting

By default, issues are sorted by impacted user count within each status group, so the most user-impacting problems always float to the top. You can change the sort order to:

- **First detected** — oldest issues first or newest issues first
- **Last updated** — recently active issues at the top
- **Priority** — strict `High → Medium → Low` ordering

### Searching

The search bar accepts free-text queries and matches against issue titles and descriptions. Searching for `password reset` will surface `#120 Broken password reset link` instantly. Search works across all status groups simultaneously.

## Subscribing to an issue

You can subscribe to any issue to receive notifications whenever it is updated — new findings from the agent, status changes, comments from teammates, or newly linked pull requests.

<Steps>
  <Step title="Open the issue card menu">
    Hover over an issue card and click the **⋯** overflow menu, or open the issue detail view.
  </Step>
  <Step title="Click Subscribe">
    Select **Subscribe** from the menu. Your avatar will appear in the subscriber list on the issue header.
  </Step>
  <Step title="Configure your notification channel">
    Notifications are delivered based on your personal notification settings — in-app, email, Slack, or Microsoft Teams. You can adjust these in **Settings → Notifications**.
  </Step>
</Steps>

<Tip>
  If you are assigned to an issue, you are automatically subscribed. You do not need to subscribe manually when an issue is routed to you.
</Tip>

## How issues appear automatically

You never create issues in Interfere manually. Interfere continuously monitors your application and detects anomalies — spikes in error rates, changes in user behavior, failed flows — without any configuration per issue type.

When Interfere identifies a new problem, it:

1. Creates the issue and places it in the **Under Investigation** group
2. Generates a title and description from the detected signals
3. Sets an initial priority based on error rate and user impact
4. Routes the issue to the appropriate owner based on your team's routing rules
5. Begins collecting evidence — sessions, logs, code changes, and deployment context

The first thing your team sees is a fully-formed issue with context, not a raw alert. For example, when the password reset link broke in production, `#120 Broken password reset link` appeared in the Inbox within minutes of the first failed reset attempt — already assigned to Luke Shiels, already flagged as a regression, already showing 2,122 impacted users.
