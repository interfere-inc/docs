When Interfere detects an issue, it doesn't just report that something went wrong — it shows you what your users actually experienced. Session replays are captured automatically for every affected user and attached to the issue without any manual instrumentation per problem. By the time your team opens an issue, a library of real user sessions is already waiting to be watched.

## What a session replay captures

A session replay is a faithful reconstruction of what a user saw and did during a single visit to your application. Each replay records:

- **User events** — clicks, form inputs, page navigations, scroll positions, and UI interactions
- **Errors** — JavaScript exceptions, network failures, and unhandled rejections, timestamped to the exact moment they occurred in the session
- **Duration** — total length of the session from first event to last
- **Visited pages** — the ordered list of URLs the user navigated through

Replays are synchronized with the event log, so you can jump to the moment an error fired and see precisely what the user was doing at that instant.

<Note>
  Session replays respect your consent configuration. If a user has not consented to session recording, their replay will not be captured or stored. See [Consent Management](/sdk/consent-management) for details on how to configure consent gating.
</Note>

## Accessing replays from an issue

Replays are available from the **Sessions** tab on any Issue Detail view.

<Steps>
  <Step title="Open the issue">
    Navigate to the Inbox and click any issue to open its detail view.
  </Step>
  <Step title="Click the Sessions tab">
    Select the **Sessions** tab below the issue header. You'll see a list of all sessions Interfere has collected for this issue, sorted by recency.
  </Step>
  <Step title="Click a session to watch it">
    Select any session card to open the replay player. Use the playback controls to watch in real time or scrub to a specific moment.
  </Step>
</Steps>

## Session metadata

Each session card in the Sessions tab displays a summary before you open the full replay:

| Field | Description |
|---|---|
| **Session ID** | A short alphanumeric identifier for the session (e.g., `j45r`, `m21x`) |
| **Event count** | Total number of events recorded in the session (e.g., *5 events*, *12 events*) |
| **Error count** | Number of errors that occurred during the session (e.g., *1 error*, *0 errors*) |
| **Duration** | How long the session lasted (e.g., *23 min*, *8 min*) |
| **Pages visited** | The pages or routes the user navigated through during the session |
| **Timestamp** | When the session occurred (e.g., *January 28, 2026 · 11:44 AM*) |

A session list for a user affected by `#120 Broken password reset link` might look like this:

| Session ID | Events | Errors | Duration | Date |
|---|---|---|---|---|
| `j45r` | 5 events | 1 error | 23 min | Jan 28, 2026 · 11:44 AM |
| `m21x` | 12 events | 0 errors | 8 min | Jan 28, 2026 · 9:12 AM |
| `q9fe` | 7 events | 0 errors | 14 min | Jan 27, 2026 · 4:38 PM |
| `b3km` | 18 events | 2 errors | 31 min | Jan 27, 2026 · 10:05 AM |

Sessions with errors are visually flagged so you can jump straight to the most relevant replays without watching every session in full.

## Reproducing user-reported problems

Session replays are one of the most reliable ways to confirm that a bug your team has debugged matches what real users actually experienced. Instead of trying to reproduce a problem from a user's description, you can watch the exact sequence of events that led to the failure.

A practical workflow for reproduction:

<Steps>
  <Step title="Filter sessions by error count">
    On the Sessions tab, sort or filter by error count to surface sessions where something clearly went wrong.
  </Step>
  <Step title="Watch the session up to the error">
    Open the session and use the event timeline on the left to jump to the moment the error fired. Watch the 10–30 seconds leading up to it to understand the user's path.
  </Step>
  <Step title="Compare with the agent's finding">
    Switch back to the **Activity** tab and compare what you observed in the replay with the agent's root cause finding. This confirms whether the proposed fix addresses the real user experience.
  </Step>
  <Step title="Use the visited pages list to reproduce locally">
    Note the pages visited and replicate the navigation path in your local environment or staging environment to test your fix before deploying.
  </Step>
</Steps>

<Tip>
  If multiple sessions show the same error at the same point in the flow, that's a strong signal you've found the consistent reproduction path. Look for the session with the highest event count — longer sessions often reveal more context about what the user tried before and after the error.
</Tip>

## Consent and replay availability

Interfere's session replay feature operates within the consent preferences your users have set. If your application uses a consent management platform such as c15t, CookieYes, or OneTrust, Interfere's SDK respects those signals automatically.

- Users who have **not consented** to session recording will not have replays captured
- Users who **withdraw consent** will have future sessions excluded; previously captured replays are handled according to your data retention policy
- The Sessions tab will indicate when an issue has affected users whose sessions were not recorded due to consent settings

For full details on configuring consent gating, see [Consent Management](/sdk/consent-management).
